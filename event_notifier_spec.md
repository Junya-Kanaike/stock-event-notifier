# 日本株イベントドリブン Slack通知システム 仕様書(MVP)

- 版: v1.0(MVPスコープ)
- 作成日: 2026-07-07
- 目的: PO・IPO・立会外分売・CB・株式分割の各イベントを自動検知し、イベント種別ごとのSlackチャンネルへ、戦略上必要なタイミングで通知する。
- 制約: **完全無料で運用**(API従量課金なし)、**無人で自動稼働**、個人利用。

---

## 1. 全体アーキテクチャ

```
┌─────────────────────────────────────────────┐
│ GitHub リポジトリ(private推奨)                │
│                                             │
│  GitHub Actions (cron)                      │
│   ├─ poll_tdnet.yml   … 10〜15分間隔で巡回    │
│   └─ daily_morning.yml … 毎営業日 07:30 JST   │
│                                             │
│  Python 3.12 スクリプト群                     │
│   ├─ collectors/  (データ取得)                │
│   ├─ notifiers/   (Slack送信)                │
│   └─ state/events.json (状態ストア, git commit)│
└─────────────────┬───────────────────────────┘
                  │ Slack Incoming Webhook (無料)
                  ▼
   #po  #ipo  #bunbai  #cb  #split  (5チャンネル)
```

### 採用理由
| 要素 | 選定 | 理由 |
|---|---|---|
| 実行基盤 | GitHub Actions | 無料(publicは無制限、privateは月2,000分)。cron対応。 |
| 通知 | Slack Incoming Webhook | 無料。チャンネル別にWebhook URLを発行するだけ。 |
| 状態管理 | リポジトリ内 `events.json` をコミット | DB不要。Actionsから読み書き可能。差分もgitで追える。 |
| 言語 | Python | requests / BeautifulSoup / pdfplumber / jpholiday が使える。 |

### 無料枠に関する注意
- リポジトリは **public で確定**(§9)。public リポジトリはGitHub Actionsの実行時間が無制限のため、分数制限を気にする必要はない。
- GitHub Actions の cron は指定時刻から**数分〜15分程度遅延することがある**。「できるだけ早く」の実効値は「開示から最大25分程度」と割り切る(MVPの許容仕様)。

## 2. データソース(すべて無料)

| # | ソース | 用途 | 取得方法 |
|---|---|---|---|
| D1 | TDnet 適時開示(yanoshin TDnet Web API `https://webapi.yanoshin.jp/tdnet/` またはTDnet閲覧サービスのHTMLスクレイピング) | PO / CB / 株式分割 / 立会外分売 / 自社株買い の開示検知 | JSONまたはHTML巡回。開示タイトルのキーワード判定 + 添付PDF解析 |
| D2 | JPX「制度信用・貸借銘柄一覧」(Excel、JPXサイトで日次公開) | 貸借/信用区分の判定 | 日次ダウンロード→ローカルキャッシュ |
| D3 | JPX「東証上場銘柄一覧」(`data_j.xls`、月次) | 市場区分(プライム/スタンダード/グロース)・銘柄名の正引き | 月次ダウンロード→キャッシュ |
| D4 | JPX「新規上場会社情報」ページ | IPOの上場日・銘柄コード・銘柄名 | 日次スクレイピング |
| D5 | JPX「立会外分売情報」ページ(補助: 開示PDF) | 分売実施日・分売条件 | 日次スクレイピング + D1のPDF |

> **実装ノート(Codexへ)**: D1はまず yanoshin API を第一候補とし、疎通不可の場合のフォールバックとしてTDnetのHTML巡回を実装する。スクレイピングは各サイト1リクエスト/巡回に留め、User-Agent明記、失敗時は指数バックオフ。

## 3. イベント別仕様

### 共通メッセージ要素
- 銘柄コード(4桁+市場拡張)、銘柄名、市場区分(D3)、貸借/信用区分(D2)
- すべての通知に `[イベント種別]` プレフィックスと開示PDFへのリンク(あれば)を付ける。

---

### ① 公募増資・売出し(PO) → `#po`

**検知条件(D1タイトルのキーワード判定)**
- 対象: 「新株式発行」「公募による新株式発行」「売出し」「株式の売出し」等を含む開示
- 除外: 「立会外分売」「自己株式の処分(のみ)」「株主割当」「行使価額修正条項」など明確にPOでないもの
- 公募 / 売出し / 両方 の区別: タイトルおよびPDF本文から判定

**A. 発表時通知(即時=次回巡回時)** — 必須項目:
| 項目 | 取得元 |
|---|---|
| 銘柄コード・銘柄名 | D1 |
| 市場区分 | D3 |
| 貸借/信用 | D2 |
| 種別(公募増資/売出し/両方) | タイトル+PDF |
| 吸収規模(億円) | PDF本文(発行価額総額+売出価額総額+OA分。概算でよい。抽出失敗時は「取得失敗」と明記して通知は止めない) |
| 発行済株式数比率(%) | PDF本文から抽出できた場合のみ記載(新株式数+売出株式数+OA ÷ 発行済株式総数) |
| 価格決定日 | PDF「発行価格等決定日」欄。「YYYY/MM/DD〜YYYY/MM/DD の間のいずれかの日」形式の場合は**期間をそのまま記載し、初日を暫定決定日として登録**(後述の決定日確定処理で上書き) |
| 想定受渡日 | PDFの払込期日・受渡期日欄から抽出。なければ「価格決定日+6営業日(暫定)」と明記 |

**B. 価格決定日の確定処理**
- 巡回で「発行価格等の決定に関するお知らせ」を検知したら、該当銘柄の決定日を**その開示日で確定**し、以後のスケジュールを再計算する。

**C. スケジュール通知(毎営業日07:30のdailyジョブで判定・送信)**
| タイミング | メッセージ |
|---|---|
| 価格決定日 当日 | 「本日は {銘柄} の価格決定日です。**寄り付きで買う**」 |
| 決定日の翌営業日 | 「本日は {銘柄} の価格決定日の翌日です。**寄り付きで買う**」 |
| 決定日の翌々営業日 | 「本日は {銘柄} の価格決定日の翌々日です。**寄り付きで買う**」 |
| 受渡日 当日 | 「本日は {銘柄} の受渡日です。**寄り付きで買う**」 |
| 決定日+25営業日 | 「本日は {銘柄} の価格決定日から25営業日後です」 |
| 決定日+26営業日 | 「本日は {銘柄} の価格決定日から26営業日後です」 |

> 「翌日」「翌々日」はカレンダー日ではなく**営業日ベース**で解釈する(決定日が金曜なら翌日=月曜)。※発注者確認事項→ §9

---

### ② IPO → `#ipo`

**検知**: D4を日次巡回し、上場予定(コード・銘柄名・上場日)を登録。

**通知(dailyジョブ)**
| タイミング | メッセージ |
|---|---|
| 上場日の前営業日 | 「明日 {銘柄コード} {銘柄名} が新規上場します」 |
| 上場日 当日 | 「本日 {銘柄コード} {銘柄名} が新規上場します」 |

---

### ③ 立会外分売 → `#bunbai`

**検知**: D1タイトル「立会外分売」+ D5。発表当日の巡回で即時通知。

**A. 発表時通知(即時)** — 銘柄コード、銘柄名、市場区分、貸借/信用、分売実施日(予定)。
- 実施日が「YYYY/MM/DD〜MM/DD のいずれか」形式の場合は期間で記載し、実施日確定開示(または前日のD5)で確定させる。

**B. スケジュール通知(dailyジョブ)**
| タイミング | メッセージ |
|---|---|
| 実施日の前営業日 | 「明日 {銘柄} の立会外分売が実施予定です」 |
| 実施日 当日 | 「本日 {銘柄} の立会外分売実施日です。**寄り付きで買う**」 |
| 実施日+5営業日 | 「{銘柄}(分売): 本日実施日から5営業日後です。**売却する**」 |

---

### ④ 転換社債(CB) → `#cb`

**検知条件(AND)**
1. D1タイトルに「転換社債型新株予約権付社債」(CB)の発行決議を含む
2. 当該銘柄が**貸借銘柄**である(D2)
3. **自社株買いを同時発表していない**:
   - 同一銘柄・同一開示日(当日中)のD1開示に「自己株式の取得」「自己株式取得」を含む開示が**存在しない**、かつ
   - CB開示のタイトル/本文に自己株式取得(いわゆるリキャップCB)への言及が**ない**
   - 判定タイミング: CB検知後、**当日20:00の最終巡回まで通知を保留せず即時通知し、その後同日中に自社株買い開示を検知したら「⚠️ 取消: 自社株買い同時発表を確認」と訂正通知を送る**(即時性と正確性の両立)

**通知内容**: 銘柄コード、銘柄名、市場区分、貸借であること、発行額(取れれば)、開示PDFリンク。

---

### ⑤ 株式分割 → `#split`

**検知**: D1タイトル「株式分割」。PDFから**分割比率**(1:N)と**効力発生日**を抽出。
- 抽出失敗時は比率・効力発生日を「要確認」として登録し、通知自体は行う。

**通知(dailyジョブ)**
| タイミング | メッセージ |
|---|---|
| 効力発生日の前営業日 | 「明日 {銘柄コード} {銘柄名}({市場区分})の株式分割(1:{比率})の効力発生日です」 |
| 効力発生日 当日 | 「本日 {銘柄コード} {銘柄名}({市場区分})の株式分割(1:{比率})の効力発生日です」 |

> 注: 発表時の即時通知は①③④のみ。⑤は発表時通知不要(効力発生日前後のみ)。※発注者確認事項→ §9

## 4. 営業日計算

- ライブラリ: `jpholiday`(日本の祝日)
- 休業日 = 土日 + 祝日 + 年末年始(12/31〜1/3)
- 関数仕様:
  - `is_business_day(date) -> bool`
  - `add_business_days(date, n) -> date`(n営業日後)
  - `next_business_day(date)` / `prev_business_day(date)`
- すべての日付処理は **JST(Asia/Tokyo)** 固定。GitHub ActionsはUTCなので変換必須。

## 5. スケジューラ設計(GitHub Actions)

### Job 1: `poll_tdnet`(即時検知系: ①発表 ③発表 ④)
- cron: `*/10 23-10 * * 0-4`(UTC)= JST 8:00〜19:59 の10分間隔、平日
- 処理: D1巡回 → 新規開示のキーワード判定 → 該当すればPDF取得・解析 → 即時Slack通知 → `events.json` 更新・コミット
- 重複防止: 開示ID(TDnetの開示番号)を `notified_ids` に記録し、既通知はスキップ

### Job 2: `daily_morning`(スケジュール通知系: ①C ② ③B ⑤)
- cron: `30 22 * * 0-4`(UTC)= JST 平日 07:30
- 処理:
  1. 当日が営業日でなければ終了
  2. D2/D3/D4/D5 の日次更新
  3. `events.json` の全イベントについて「本日が通知対象日か」を営業日計算で判定し、該当メッセージを送信
  4. 送信済みフラグを立てて再コミット

## 7. Slack通知仕様

- チャンネル別に Incoming Webhook を5本発行し、GitHub Secrets に登録:
  - `SLACK_WEBHOOK_PO` / `SLACK_WEBHOOK_IPO` / `SLACK_WEBHOOK_BUNBAI` / `SLACK_WEBHOOK_CB` / `SLACK_WEBHOOK_SPLIT`
- 加えて `SLACK_WEBHOOK_SYSTEM`(任意): ジョブ失敗・PDF解析失敗などの運用アラート用
- メッセージはSlack Block Kit(header + fields)で整形。アクション指示(「寄り付きで買う」等)は**太字**。
- 送信失敗時は3回リトライ(指数バックオフ)、それでも失敗ならジョブをfailさせてActionsのメール通知に頼る。

### 発表時通知の例(PO)
```
📢 [PO発表] 7203 トヨタ自動車(プライム / 貸借)
種別: 公募増資+売出し
吸収規模: 約1,200億円(発行済比 8.3%)
価格決定日: 2026/7/15〜7/17 のいずれか(暫定: 7/15)
想定受渡日: 2026/7/23(暫定)
開示PDF: <link>
```

## 8. エラー処理・運用方針

1. **通知は「欠落より過剰」を優先**: 解析に失敗しても、判明している情報だけで通知を出し「⚠️ ○○の抽出に失敗」と付記する。
2. PDF解析(pdfplumber)は正規表現ベース。書式ゆらぎで失敗し得る前提で、失敗ログを `SLACK_WEBHOOK_SYSTEM` に送る。
3. データソースのHTML構造変更検知: 巡回で0件が3営業日連続したらシステムアラート。
4. `events.json` のコミット競合: Job1とJob2が同時実行されないようcronをずらし、`concurrency` グループで直列化。
5. すべての外部アクセスにタイムアウト(15s)とUser-Agent設定。

## 9. 仕様確定事項(発注者確認済み・2026-07-07)

以下はすべて発注者に確認済み。**Codexは再確認不要、この通り実装すること**:

1. **PO「翌日・翌々日」の解釈**: **営業日ベースで確定**(決定日が金曜なら翌日=月曜)。
2. **PO 25/26営業日後の起点**: **価格決定日を0日目として +25 / +26営業日で確定**。
3. **CBの通知方式**: **即時通知 → 同日中に自社株買い開示を検知したら訂正通知(§3-④)方式で確定**。
4. **株式分割**: 発表時の即時通知は**不要**。効力発生日の**前営業日と当日のみ**通知で確定。
5. **リポジトリは public で確定**。Actions実行時間は無制限のため、巡回間隔は**10分**とする。Slack Webhook URLその他の秘匿情報はGitHub Secretsのみに保存し、コード・events.json・ログへのハードコードを厳禁とする。

## 10. MVPスコープと将来拡張

## 12. 受け入れ基準(Doneの定義)

- [ ] 過去のTDnet開示サンプル(PO/CB/分割/分売 各3件以上)でキーワード判定の単体テストが通る
- [ ] 営業日計算が祝日・年末年始・連休またぎでテストを通る
- [ ] ダミーイベントを events.json に手動投入し、daily ジョブが正しい日に正しい文面を送る
- [ ] 同一開示IDで二重通知が発生しない
- [ ] GitHub Actions 上でSecretsのみで動作し、コード内にURL/トークンのハードコードがない
